# 技术栈与架构方向

## 固定选择

- **框架**：React Native（具体是否采用 Expo 在落地时选定；若选 Expo，可天然带 Web/H5 出口）。
- **语言**：TypeScript（**`strict` 必须开启**，见根 `tsconfig.json`）。
- **范围**：纯前端；状态管理、导航、本地存储选型在实现阶段确定，须保持可测试、可 tree-shake。

## AI 友好选型（与代理协作）

与 Cursor / 自动化代理协作时的推荐组合见 **`06-ai-coding-rn.md`**，摘要如下：

- **国际化**：i18next + react-i18next（字符串集中、可类型化）。
- **样式**：Tailwind + NativeWind 与 Web/H5 对齐；纯 RN 侧也可用 StyleSheet，分层避免混用失控。
- **质量**：ESLint + Prettier；目录保持扁平、单文件职责单一。

## 多端策略

| 形态 | 用途 |
|------|------|
| iOS / Android | 主产品；商店计费、推送、原生能力。 |
| H5 | 轻量访问、分享链接、**PR 预览**；与原生共享业务模块时需抽象平台差异（存储、支付、通知）。 |

## 建议分层（实现时遵循）

1. **领域与用例**：与 UI 无关的纯 TS 模块（习惯记录、订阅状态机占位等）。
2. **UI**：组件与屏幕；RN 可用 StyleSheet 或 NativeWind（见 `06-ai-coding-rn.md`）；Web/H5 侧样式与类名约定见 `05-conventions.md`。
3. **适配层**：`*.native.ts` / `*.web.ts` 或统一 `Platform.select`，避免在业务组件中散落平台判断。

## 依赖与安全

- 仅引入维护活跃、许可证清晰的依赖。
- 第三方分析 / 崩溃上报若接入，须在隐私政策与用户可见位置说明。

## 与「无代码」阶段的关系

本文档描述**目标态**；具体 `package.json`、目录结构在首次提交应用代码时建立，并在此文件或 `README` 中更新「实际选型」小节。
