# 技术栈与架构方向（WHAT / WHY / HOW）

## WHAT：固定边界

- 开发框架：Expo（底层能力基于 React Native）。
- 语言：TypeScript，且 `strict` 必须开启。
- 范围：纯前端，不引入自建后端业务 API。
- 平台：iOS、Android、H5（仅用于 PR 预览）。

## WHY：关键决策原因

- TS strict 提升可维护性，减少 AI 修改导致的隐式回归。
- 纯前端边界降低复杂度与合规风险。
- 多端共用业务逻辑可提升迭代效率，但平台差异必须显式隔离。

## HOW：落地分层（建议）

1. 领域层：纯 TS 业务逻辑，不依赖 UI 或平台 API。
2. 展示层：页面与组件，RN 采用官方 `StyleSheet`。
3. 适配层：`*.native.ts` / `*.web.ts` 或 `Platform.select` 封装差异。

## 测试要求（必须）

- 测试框架统一使用 `Jest`，且仅做单元测试。
- 每个功能改动必须新增或更新单元测试，不允许“只改代码不补测试”。
- 测试文件统一放在 `src/**/__tests__` 目录下。
- 测试最小覆盖包含：成功路径、失败路径、至少一个边界条件。
- 平台差异逻辑需至少一条针对性单元测试用例。
- 提交前需在本地执行测试命令并记录结果。

## 选型要求（AI 友好）

- 国际化：`i18next` + `react-i18next`。
- 质量保障：`ESLint` + `Prettier` + 类型检查。
- 目录策略：扁平、职责单一，减少检索和修改成本。
