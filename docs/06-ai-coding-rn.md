# AI 协作指南（RN 项目）

本文件解决两个问题：让 AI 少犯错、让多人协作输出稳定一致。

## 高优先级指令

- 先读与任务相关的文档，不要全量加载所有说明。
- 改动前先确认边界：纯前端、TS strict、平台差异分层。
- 输出后对齐检查：lint、typecheck、Jest 单元测试。
- 所有功能改动必须补测试；禁止“无测试提交功能代码”。

## 推荐技术组合

- 包管理：`pnpm`（必须）
- 语言：TypeScript strict（默认禁止 `any`，例外需注释）
- 样式：React Native 官方 `StyleSheet`
- 国际化：`i18next` + `react-i18next`
- 质量工具：ESLint + Prettier
- 测试框架：Jest（仅单元测试）

## 写规则的方式（比堆规则更重要）

- 规则短句化：一条规则只表达一个约束。
- 规则可执行：尽量能映射到命令、检查或明确动作。
- “不要”后必须跟“应该”：避免代理卡住。
- 只保留高频错误相关规则，定期清理无效项。

## 推荐规则目录

- `.cursor/rules/` 下按主题拆分（如 `testing.mdc`、`rn-style.mdc`、`i18n.mdc`）。
- 使用 frontmatter `paths` 限定生效范围，降低无关规则干扰。

## 何时更新规则

- 同类错误在一周内重复出现 >= 2 次。
- 代码评审中反复出现相同偏差。
- CI 中出现可预防的固定模式失败。

## 测试用例要求（执行层）

- 每个新功能或行为变更，至少补 3 类单元测试用例：成功、失败、边界。
- 单元测试文件统一放在 `src/**/__tests__` 目录下。
- 覆盖率门槛为 `70%`（lines、branches、functions、statements）。
- 仅重构但行为不变时，需保证关键路径测试全部通过，不得删除有效断言。
- 缺陷修复必须附带回归测试，避免重复引入。
